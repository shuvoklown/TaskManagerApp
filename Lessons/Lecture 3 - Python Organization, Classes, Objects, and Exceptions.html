Object oriented programming is a style of programming in which we process data using objects. This style of programming is also found in other languages like C#, Ruby, Java, and many others. Python allows us to write code in this style as it helps organize our projects. Python, if you haven't figured it out by now, is an Object Oriented language that doesn't force us to use object oriented programming. It is very flexible in how it allows us to write code. This is what makes it very popular as anyone coming from other language paradigms (imperative, functional, logical) can get a grasp of the language with ease while programming in their style.

Before we begin, I would like to define some terms for anyone that may have forgotten.

Inheritance

Inheritance allows one class to obtain the attributes of another class. The instance variables and methods become part of the subclass.

Classes

Classes are like blue prints, they define what an instance of the class can do and what data it stores. These instances are called objects.

Objects

Objects are instances of classes. The class states what an object can do with method definitions and what an object can hold.

Attributes

All methods and instance variables are called attributes in Python.

Methods

Methods are functions defined inside of a class that only objects of that class can use.

Instance Variables

These are variables that belong to instances of a class and can contain different values for each instance.

Class Variables

In other languages, these would be considered static variables. These variables belong to the class and are not unique for each instance of the class. You don't even need an object of the class to use these.

Access Modifiers

An access modifier tells a compiler where an attribute can and can't be read from or written to. Usually, there are three kinds of access modifiers Public, Private, and Protected. Python only has one, Public. You can access anything and everything from a Python class. It is up to the programmer to be responsible.

Interface

Interfaces contain empty method bodies. Interfaces give a common ground between multiple classes by creating a contract stating that you must implement these methods. When you implement an interface in a class you are guaranteeing any instance of that class to contain those sets of methods. Interfaces can have object instances of them created.

Abstract Class

Abstract classes are a bridge between classes and interfaces allowing you to define methods with a body or defining abstract methods that others must implement. Abstract classes can't have instances of them created.

Super Class/Parent Class/Base Class

A super class, parent class, or base class is the class that a subclass inherits from.

Subclass/Child Class

A subclass or child class is a class that inherits from a super class.

Overriding

Overriding happens when a method with the same name and number of parameters get's a different body. This is common for subclasses to do when they want to redefine what a method the super class has to do something specific to them.

Overloading

Overloading happens when we have a method defined multiple times in the same class. Usually, this occurs when we have a method that could take a different number of arguments or arguments of different types, however, with Python's default variables and dynamic typing, we don't have to do this. We can define a single method with all the variables we would like and if we have to set some variables to a default value.

Lecture 3.1.0: Class Variables and Instance Variables

There is a major difference between class variables and instance variables. Class variables belong to all instances of the class while instance variables belong to the instance. This ultimately means that if a class variable is changed using the class to change it, it is changed for all instances while if an instance variable is changed it is only changed for that instance of the class.

Some of this code may not be familiar but for now just notice what happens to the variables when an instance changes it to when the class changes it.

class Test:
    class_variable = 3
        
t1 = Test()
t2 = Test()

print("T1 Class Variable:", t1.class_variable)
print("T2 Class Variable:", t2.class_variable)

Test.class_variable = 1000

print("New T1 Class Variable:", t1.class_variable)
print("New T2 Class Variable:", t2.class_variable)

T1 Class Variable: 3
T2 Class Variable: 3
New T1 Class Variable: 1000
New T2 Class Variable: 1000

Notice that when the class variable gets changed it changes it for all instance variables. There is a slight gotcha, however, because you can shadow a class variable with an instance variable. Let us change class_variable using an instance of the Test class such as t1.

t1.class_variable = 5000

print("Test Class Variable:", Test.class_variable)
print("New T1 Class Variable:", t1.class_variable)
print("T2 Class Variable:", t2.class_variable)

Test Class Variable: 1000
New T1 Class Variable: 5000
T2 Class Variable: 1000

Notice now that only t1 has class_variable changed to 5000. This is because t1 creates an instance variable called class_variable that shadows the actual class variable class_variable. We can get back to the original class variable by using the del keyword.
(WARNING: del should be used sparingly. If you delete a variable that is used later in the program then your program will crash)

print("Test Class Variable:", Test.class_variable)
print("T1 Class Variable Before Delete:", t1.class_variable)

del t1.class_variable

print("T1 Class Variable After Delete:", t1.class_variable)

Test Class Variable: 1000
T1 Class Variable Before Delete: 5000
T1 Class Variable After Delete: 1000

Lecture 3.1.1: Class Methods

Methods in a class are just functions but must have an object to be able to use the method. There is one thing that is different, however. That is, methods must always have one argument defined that is the reference to the instance. When the method is called, the Python interpreter passes the instance of the class to the method as the first argument behind the scenes. This first argument can then be used to call other attributes of the class like other methods or class or instance variables. We can also have local variables inside the method but they will get removed from memory once the method is done and has returned to the original call point. This first argument that is a reference to the instance of the class can be named anything you want but the conventional name is to use self.

class Test:
    def __init__(self):
        self.stays_around = "I am alive"
    def print_stuff(self):
        will_get_removed = "I am alive, for now"
        print(will_get_removed)
        
t = Test()
print("t.stays_around ->",t.stays_around)

t.print_stuff()

print(t.will_get_removed)

I am alive, for now

This code demonstrates the difference between an instance variable and a local variable. A local variable belongs to the block it was created in and will be removed from memory by the garbage collector when the block is done. In this case, will_get_removed is a local variable to print_stuff method. Believe it or not, but self is a local variable to both print_stuff and __init__. (In other words, the self in __init__ and print_stuff will live only as long as the method scope).

When you call the method, you DO NOT need to provide an argument for that first argument. So, if the method doesn't accept any other arguments you use it like a normal method by using the dot operator. If the method does accept additional arguments, you will be asked to pass in those arguments. You will also get an error that looks like an off-by-one error, something like "method expects 2 arguments, 1 given", when you call a method that expects another argument other than the instance and you call it with nothing.

Might be confusing, but you will get used to it.

Using that first argument, we can access other attributes of the instance object.

class Test:
    def __init__(self):
        self.x = 3
    def getX(self):
        return self.x
    def setX(self, newX):
        self.x = newX
        
t = Test()
print("t.getX() ->", t.getX())

t.setX(500)

print("t.getX() ->", t.getX())

t.getX() -> 3
t.getX() -> 500

Here, I defined a class that will create a variable, x, for all instances of the class. Then, I define getters and setters (which aren't required in Python because everything is Public)*. Inside these methods I used self to access the instance variable x. Notice, in setX, I have self and newX. When I call this method, I must give it an argument that will be set to newX which is a local variable to the setX method.

The reason for the first argument, self, that is a pointer to the instance is to differentiate between local variables and instance variables. If we didn't have this, Python would mistake the variable as a local variable and it would be removed from memory as soon as the method is done running.

* It is actually Pythonic not to have getters and setters and to just access the variable using the dot operator. This was just to demonstrate methods and the self argument.

Lecture 3.1.2: Class Constructor

Python has an interesting way to do constructors. We actually use a method (discussed more in section 3.2) called __init__ that runs when an object is created. (There is another, __new__ but is not discussed here). When we create an object, behind the scenes, the __init__ method is being called. We use this method to instantiate (create and assign) variables for an object. If we include other arguments for __init__ besides self we will need to make sure we pass arguments when needed. We can, of course, use default arguments, keyword arguments, and all the other argument types since __init__ is just like a normal method.

class Person:
    def __init__(self, name, age=0):
        self.name = name
        self.age = age
        
p1 = Person("Sammy")
p2 = Person("Brian", 27)

print("P1's name is {} and age is {}".format(p1.name, p1.age))
print("P2's name is {} and age is {}".format(p2.name, p2.age))

P1's name is Sammy and age is 0
P2's name is Brian and age is 27

Here, I created a class with an __init__ method that accepts 2 additional values for name and age. I then set instance variables of the same name as the local variables to the values pointed at by the local variables. One of these variables is a default variable and so is optional when creating an object from the class. I then created 2 objects with different names and ages. This is the use for constructors.

You don't have to define the __init__ method for classes. It is optional but is useful for creating default variables that are used throughout the classes methods.

Lecture 3.1.5: Inhertiance

Inheritance allows us to define methods and variables in one class and have another class utilize those same attributes without having to define them itself. If you have a method in one class, for example, and another class inherits that class, then the subclass has access to that method. It can then redefine that method to do whatever or keep its default behavior. This is great for collections but even better if the action/method does the same thing for any subclass inheriting from the super class giving the option to change how the method works if need be.

To have a class inherit from another class, we put the super class inside the parenthesis' when defining the class name. You have already seen this with the object class. Let's take a look at another class.

Here is an example using animals. (You may want to enter this in a script file for ease in case of syntax errors)

class Animal():
    def __init__(self, name):
        self.name = name
        
    def talk(self):
        return "NO SOUND"
    
class Bacteria(Animal):
    def __init__(self, name):
        Animal.__init__(self, name)
        
class Duck(Animal):
    def __init__(self, name):
        Animal.__init__(self, name)
        
    def talk(self):
        return "QUACK"
        
class Dog(Animal):
    def __init__(self, name):
        Animal.__init__(self, name)
        
    def talk(self):
        return "BARK"
        
class Daffy(Duck):
    def __init__(self):
        Duck.__init__(self, "Daffy")
    
    def talk(self):
        
        return super().talk() + ": You're dethpicable."

bact = Bacteria("Amoeba")
duck = Duck("Mallard")
dog = Dog("Shadow")
daffy = Daffy()

print(bact.talk())
print(duck.talk())
print(dog.talk())
print(daffy.talk())

NO SOUND
QUACK
BARK
QUACK: You're dethpicable.

Here, we have five classes. A class Animal which defines a talk method and a name variable. The subclasses of Animal, Bacteria, Duck and Dog, define their own talk method. Daffy inherits from Duck and thus has a talk method that would have returned "QUACK" but is defined to return something else. Daffy class does, however, not need an argument passed when creating an object for it as it passes a name to the super class for us with a name. The use of super() allows us to access the super class and any of it's variables or methods.

Why would any of this be necessary? As programmers, you write code that takes advantage of collections of data, whether it be arrays, lists, tuples, dictionaries, sets, or trees, if you access an object from that collection you want to be sure that object has a method or else it will throw an exception. Now, modify the code a bit to show you what I mean.

list_of_animals = [duck, dog, daffy, bact]

for animal in list_of_animals:
    print("{} says '{}'".format(animal.name, animal.talk()))

Now when it runs, assuming all elements of the list are animals (they are) they will have their version of the talk method called and have their name displayed. We can add a step further and use isinstance to check if they are in fact an instance of Animal. DON'T USE type() TO COMPARE INSTANCE TYPES. The type function will only return an objects immediate object type. isinstance will do a deep dive and will actually check for all types.

For those of you who saw what I did but didn't mention, Animal is actually an Interface. I can create an object of Animal and call the talk method but an error will be raised. Python doesn't really have a distinction between an interface or an abstract class like other languages (C#, Java, C++). If you need an interface, you create one. If you need an abstract class, define the class that way.

Lastly, you can do multiple inheritance (Most other object oriented languages don't allow this.) I don't recommend it because if you define a method with the same name and arguments in two different base classes and then subclass those classes in one class, you will have a conflict with that method. (Which method get's ran when it's called?) I recommend sticking to single inheritance (or you can read about composition vs inheritance).

Lecture 3.2: Object Data Model

The object data model is Python's solution to data representation. Along with this representation are methods, sometimes called magic methods, dunder (for double underscore) methods, or more correctly data model methods that allow us to determine how an operation is to handle the object. You've already seen one such method, the __init__() method. Another example is the __add__() method which is called whenever you use the + operator (even on numbers). Here is an exmaple of what is happening behind the scenes using strings concatenation.

"Hello, " + "world!" #What you use.

'Hello, world!'

"Hello, ".__add__("world!") #What is happening behind the scenes.

'Hello, world!'

For numbers, we can do the same thing but since the number symbol is a literal we first need to allow the interpreter to create the number object and then call the method. Here is what you might try at first.

2.__add__(3)

  File "<ipython-input-3-4ba1c06ab832>", line 1
    2.__add__(3)
            ^
SyntaxError: invalid syntax

As you can see, we get a SyntaxError. We need to surround the number calling the method with parenthesis to encompass the object and allow it to be created first. Here is the fix.

(2).__add__(3)

5

Alternatively, you can do this when the number is referenced by a variable.

t = 2
t.__add__(3)

5

This is what is happening when you run 2 + 3 or t + 3.

Now you should have an understanding of what is so magical about these methods. We can define them for our classes to behave a certain way when an operation, or built-in function, is called on our objects. Here is a list of data model methods that are common but you can find more at the link provided.

__str__: This data model method should return a string that represents the contents of a class. When print is given an object of a class it class that objects __str__(self) method and displays the string to the output.

__add__: This data model method get's called during an add operation (3 + 2 for instance). The object on the left hand side is the object that will call the method with the object on the right being passed in as an argument. We can define a behavior for our own objects when they are on the left hand side of the add operation.

__sub__: This data model method get's called during a subtraction operation (3 - 2 for instance). The object on the left hand side is the object that will call the method with the object on the right being passed in as an argument. We can define a behavior for our own objects when they are on the left hand side of the add operation.

__mul__: This data model method get's called during a multiplication operation (3 * 2 for instance). The object on the left hand side is the object that will call the method with the object on the right being passed in as an argument. We can define a behavior for our own objects when they are on the left hand side of the add operation.

__truediv__: This data model method get's called during a division operation (3 / 2 for instance). The object on the left hand side is the object that will call the method with the object on the right being passed in as an argument. We can define a behavior for our own objects when they are on the left hand side of the add operation.

__pow__: This data model method get's called during a power operation (3 ** 2 for instance). The object on the left hand side is the object that will call the method with the object on the right being passed in as an argument. We can define a behavior for our own objects when they are on the left hand side of the add operation.

__cmp__: This data model method get's called when the compare built-in function is called to compare two objects.

__lt__: This data model method get's called during a less than operation (3 < 2 for instance). The object on the left hand side is the object that will call the method with the object on the right being passed in as an argument. We can define a behavior for our own objects when they are on the left hand side of the add operation.

__gt__: This data model method get's called during a greater than operation (3 > 2 for instance). The object on the left hand side is the object that will call the method with the object on the right being passed in as an argument. We can define a behavior for our own objects when they are on the left hand side of the add operation.

__le__: This data model method get's called during a less than or equal to operation (3 <= 2 for instance). The object on the left hand side is the object that will call the method with the object on the right being passed in as an argument. We can define a behavior for our own objects when they are on the left hand side of the add operation.

__ge__: This data model method get's called during a greater than or equal to operation (3 >= 2 for instance). The object on the left hand side is the object that will call the method with the object on the right being passed in as an argument. We can define a behavior for our own objects when they are on the left hand side of the add operation.

Any other operations you can think of (like slicing, indexing (both sequences and mappings), comparison operators, etc.) can be overridden.

Here is an example using vectors from Physics.

class Vector2D(object):
    '''Vector2D object with an x and y value.'''
    def __init__(self, x=0, y=0):
        self.X = float(x)
        self.Y = float(y)
        
    def __add__(self, other):
        if isinstance(other, Vector2D):
            newX = self.X + other.X
            newY = self.Y + other.Y
            return Vector2D(newX, newY)
        else:
            raise ValueError("Can only add a vector to another vector.")
            
    def __sub__(self, other):
        if isinstance(other, Vector2D):
            newX = self.X - other.X
            newY = self.Y - other.Y
            return Vector2D(newX, newY)
        else:
            raise ValueError("Can only subtract a vector to another vector.")
            
    def __mul__(self, scalar):
        if isinstance(scalar, (int, float)):
            newX = self.X * scalar
            newY = self.Y * scalar
            return Vector2D(newX, newY)
        else:
            raise ValueError("Can only multiply a vector to a scalar.")
    
    def __truediv__(self, scalar):
        if isinstance(scalar, (int, float)):
            newX = self.X / scalar
            newY = self.Y / scalar
            return Vector2D(newX, newY)
        else:
            raise ValueError("Can only divide a vector to a scalar.")
            
    def __str__(self):
        return "(X = {}, Y = {})".format(self.X, self.Y)
        
        
v1 = Vector2D(2, 3)
v2 = Vector2D(7, 2)

print("Vector 1: {}".format(v1))
print("Vector 2: {}".format(v2))
print("V1 + V2 = {}".format(v1 + v2))
print("V1 - V2 = {}".format(v1 - v2))
print("V1 * 3.14 = {}".format(v1 * 3.14))
print("V2 / 2 = {}".format(v2 / 2))
print("Vector 1: {}".format(v1))
print("Vector 2: {}".format(v2))

Vector 1: (X = 2.0, Y = 3.0)
Vector 2: (X = 7.0, Y = 2.0)
V1 + V2 = (X = 9.0, Y = 5.0)
V1 - V2 = (X = -5.0, Y = 1.0)
V1 * 3.14 = (X = 6.28, Y = 9.42)
V2 / 2 = (X = 3.5, Y = 1.0)
Vector 1: (X = 2.0, Y = 3.0)
Vector 2: (X = 7.0, Y = 2.0)

In the code above, I overrode the addition, subtraction, multiplication, and division operators to allow two vectors to be added together making sure to do my own checks and raise a value error with my own message. You could just let it go and throw an error when the code tries to access attributes from other objects that might not have those attributes but you will get an AttributeError saying so and can end up being confusing when trying to debug the code.

Notice, I did not need to explicitly call those methods. The operators did that for me. Also, the print function returned what was created and returned by the __str__ function.

Lastly, you do not need to return anything when writing these methods. I just did to make my class pseudo-immutable. You could have modified the vector, however, this brings up the question of which one should be modified? Should the left vector or right vector operand be changed? I don't think either should be modified and should only change if an assignment is part of the operation for something like this, v1 += v2 (there is a data model method for this operator +=). The Vector class is also only pseudo-immutable because you can still access X and Y with the instance objects (v1.X = 3 for instance ((no pun intended))). There is a way to make it completely immutable so instance variables can't be changed but I leave that as an exercise for you.

Lecture 3.3: Polymorphism

Polymorphism looks like a big fancy word but all it means is many forms or many types. Polymorphism is what makes Object Oriented Programming so powerful. We can have objects be considered many types which will allow us to perform many operations on many objects within a collection without worrying about it needing to be a specific type. An example of this, consider a video game where you have different types of enemies. A good example is any RPG (role playing game) with different races. The game may contain a human, orc, and dwarf race which may have a class of the same name. These races can fall under a Humanoid (human-like) class. This class can fall under a Race class which can fall under an Actor that falls under a GameObject class. We can see the hierarchy like so

Looking at the hierarchy, you can see that Human, Orc, and Dwarf, while separate classes, are considered Humanoid. So, anything a Humanoid can do all three can do. Humanoid and Creature are both of type Race so they both can do whatever a Race can do. Race is a type of Actor and Actor is a type of GameObject. So, you can also say that Orc, Human, and Dwarf are of type GameObject, Actor, and Race. What they can do and what data they can store is decided by the methods and instance variables and the access modifiers for them.

The animal class in the section Lecture 3.1.5: Inhertiance has code that shows basic polymorphism in that any Daffy object is also a Duck and an Animal.

Lecture 3.4: Exception Handling

I will be using errors and exceptions interchangably through out this lecture. This is because most exceptions we catch will have Error appended to it. Technically, exceptions occur during runtime while the only real errors in Python are syntax errors. Other languages that utilize exceptions have checked and unchecked exceptions. Python only has unchecked exceptions which occur at runtime so they are just exceptions.

Python uses exceptions to handle all sorts of coding situations (even some you might not consider). For instance, generators, when they have no more elements and a next is called, will throw a StopIteration error. When you press CTRL+C to stop a program, the Python interpreter will raise a KeyboardInterrupt error to stop the execution. You can actually catch this error and print a more user-friendly message if you wanted to. When should you raise exceptions? That's really up to the programmer.

Lecture 3.4.1: Try...Except

To catch exceptions, we use the try keyword for to try out some code. If that code raises an exception then we must catch it using the except keyword. We can use multiple except statements for as many exceptions as we would like to catch. I recommend using an except statement for any KNOWN exceptions that may be thrown and a final general purpose except that is a catchall for anything else bad that might happen that you weren't ready for. The general one should give more information about the state of the program than any other exception so debugging is much easier.

try...except clauses allow us to catch errors that, when thrown, stop the program. This is also known as a crash. Catching these will allow the user to continue using our program without needing to restart it. You can say that try...except clauses are a form of control flow for error handling. There is another part which is else which can be used for clean up such as closing a file or socket or giving a resource back. The else is optional but is shown here to give you a complete example.

Let's look at an example.

file_h = None
try:
    file_h = open("fergle.in", "r")
except:
    print("Error opening file")
else:
    file_h.close()

Error opening file

Here, I have a program that tries to read a file. That file is not located where ever I started the interpreter at and so the except clause catches the error thrown and displays 'File not found'.

The exception thrown can be seen when we remove the try...except...else. The else runs if no error is thrown so the file can be closed if it is opened. Otherwise, there was an error. Whether it doesn't exist or we don't have permissions or some other problem we catch it all (which is bad programming to have a catchall like this out the gate). How can we determine what to catch then? Well, let's get rid of the try...except...else altogether.

open("fergle.in", "r")

NOTE: The general catchall excpet clause must come last, otherwise, it will always be thrown for every exception raised.

What if we want to reference the exception object raised when an error occurs? We can do that by assigning a variable.

file_h = None
try:
    open("fergle.in", "r")

    print(err)
except:
    print("Something really bad happened")

[Errno 2] No such file or directory: 'fergle.in'

We use the as keyword to create an alias. This makes sense since as is used to create aliases in other situations (such as with open("file.in", "r") as in_file).

BaseException will catch any and all other exceptions that inherit from it (all built-in exceptions extend BaseException) and ZeroDivisionError is a built-in that is more specific. You will always want to list the specific exceptions FIRST before the catchall as you will have more control of your code, can display different custom messages for each error, and other people looking at your code won't want to hunt you down 10 years later (and you won't want to hunt yourself down 2 months later when you try to read your own code).

Python does allow you to catch multiple exceptions on the same except line if you choose to do so. To do that you can separate the types of exceptions with a comma and surround the exceptions with a parenthesis (you are effectively creating a tuple of exception types).

Type this in a script file before trying it out

# FILENAME: exceptions.py
from sys import exit

while True:
    try:
        input("Press ENTER to open file")
        open("fergle.in", "r")
    except (IOError, KeyboardInterrupt) as err:
        if isinstance(err, IOError):
            print("File not found")
        elif isinstance(err, KeyboardInterrupt):
            print("\nExitting program...")
            exit()
    except EOFError:
        pass

Here, I had to catch an extra error, EOFError, if a KeyboardInterrupt is raised. The keyword pass is just a nice way of saying, "I'll create the body for this block code later so don't yell at me". If you remove the pass keyword, it throws a syntax exception because the block code (any block code) doesn't have a body. It's good for prototyping and designing when you just don't want to write implementations right now and get your codebase designed.

Lecture 3.5: Python Organization

As mentioned in earlier lectures, Python has a hierarchy that it uses to organize it's code. At the top we have packages, also called libraries. Packages contain modules. Modules are our Python scripts and other packages. Then we have classes and functions. Inside those classes and functions, we have statements and inside of statements are expressions. each layer, down to functions, is importable.

To create a package, we must include a __init__.py file. It doesn't need to contain anything, it just needs to be there. Think of your project directories. Assuming you aren't using an IDE that automatically creates the __init__.py file automatically, these folders are not packages. You will not be able to import them. If you add an __init__.py file in them, they become a package. This is also true for any folder inside a package. If the folder does not have the file it is not a package. So, if you are making a library for others to use, you must include __init__.py for any folder you want to be a package.

Why is this important? So you know how you should organize your code but also know how to import something. When you import, you might need to go deep into the package system to get the code you want.

Also, the __init__.py file does not need to be empty. It can contain code that does error checking and validation before allowing a package to be imported.

This short blog post is a good example of what the __init__ file can do for you.

An example hierarchy of the file system for a package might look like this:

Codec
|--__init__.py
|--MP3
   |--__init__.py
   |--convert.py
|--WAV
   |--__init__.py
   |--convert.py

Here, we have a package Codec that contains other packages MP3 and WAV which contain their own versions of convert.py. This might be one way to organize this project, whatever it may be, but know it is not the only way.

Lecture 3.6: Importing

Importing in Python is very powerful. We can import whole packages, individual script files in a package, individual classes from a script file, a single function in a script file, or even just a variable defined inside the script. In order to do this, you need to understand the parts of the import statement (and how many ways there are to import something)

Let us start with a simple script file.

# FILENAME: module.py
class Class(object):
    def method(self):
        return "Method called"

def func():
    return "Function called"
    
variable = "I am here."

Save this code to a script file somewhere and open your interpreter at the location you saved the file to.

To import something, we use the import keyword. You do not need to import at the top of your script files, by the way. You can import anywhere a statement can be executed (so not in a comment). I like keeping my imports at the top. You can even perform some check using an if statement and import code based on the result. This is done a lot for projects that can use either built-in modules or a preferred 3rd part module.

Next comes the thing we want to import. If we wanted to import the whole script file we simply give the name of the file without the .py.

>>> import module

Then we can use everything inside of the script file such as the class, function, and any variables by using the dot operator which syntactically looks like this. caller.callable

>>> import module
>>> c_obj = module.Class()
>>> c_obj.method()
'Method called'
>>> module.func()
'Function called'
>>> module.variable
'I am here.'

But what if we just wanted the class or function or variable? Instead, we use the keyword from and write it almost like English.

>>> from module import func
>>> func()
'Function called'
>>> module.variable
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
NameError: name 'module' is not defined

Notice, when we used func() it acted correctly but when I tried to access the modules variable it threw an exception. This is because func is loaded in the symbol table with the code that is defined for func and module.variable is not. Module isn't even defined.

We can import multiple things from the same module or package by separating them using a comma in a from statement.

>>> from module import func, variable
>>> func()
'Function called'
>>> variable
'I am here.'
>>> c_obj = module.Class()
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
NameError: name 'module' is not defined

Here, I imported the function and variable from module but the Class was not imported resulting in the exception shown.

The last bit of syntax you can use with an import statement is the as keyword. The as keyword can be used to create an alias of the module you are importing.

>>> import module as mod
>>> mod.func()
'Function called'

This is useful for shortening module names in your code.

There is a way to import all the items without importing them separately using an asterisks. For example, from itertools import *. There is a problem with this, however. One example is in the module os which contains a function open(). This is not the same function as the built-in function that we use to open files. So, if you import the os modules open() using an asterisk it will shadow the built-in open() and will run differently than you may want. It will also take up more space since you are importing everything into memory, even if you don't use it. I recommend explicitly importing the items you want so there is no confusion how you want your code to work.

Lastly, when you are creating a package or class or function or variables, you don't want to include a period. Variables, functions, and classes won't have this problem as that is against the rules of naming but packages can have this problem. If you do this, and then try to import the package, the interpreter will think you are trying to import another package or module and will throw an error because it can't find it. Instead, change the name of your package to either have an underscore or don't have anything in between.

Lecture 3.7: if name == "__main__"

What does this line of code check for? if __name__ == "__main__":? Well, every program needs a starting point of execution. Python starts it's point of execution at the top of the script file and, going from top to bottom, executes the code. When you import a module, the import system will do the same thing. So if you have statements of code that are not within a function or class they will execute. If you don't want the code to execute upon importing you can add this line and put everything under its block that you don't want to run when importing the file. The only way it will run is when the script file is called directly from the command line by passing it to the python command. Here is an example.

# FILENAME: main.py
print("Super Secret Stuff")

>>> import main
Super Secret Stuff

After importing the main module it ran the print statement. Let's make it so it won't run unless the script file is called directly.

# FILENAME: main.py
if __name__ == "__main__":
    print("Super Secret Stuff")

>>> import main
>>>

As you can see, nothing happened. Now let's call the file from the command line.

C:\Users\broge2\Desktop>python main.py
Super Secret Stuff

It ran the print statement.

This is useful for when you are creating a module with many classes and you want to write some test code within the same module. You can then test your module by calling it directly and having it run the test code. Then, when you import the module elsewhere in your code, it won't run the test code. Once you are confident your code is working as it should, remove the if __name__ == "__main__" block.
